SENSUS AMR ROW-SHIFT SPIKE -- POST-SKYSPARK SANITIZATION FIX
============================================================

Every month the Sensus AMR system exports AMRoutputYYYYMMDD.csv and SkySpark's
CSV Importer pairs consecutive snapshots to compute monthly usage deltas. When
the CSV row order shifts between months (accounts dropped, added, or re-routed),
SkySpark crosses building identities and produces impossible "usage" values in
the billions of gallons. The Feb 2026 export nulled by this effect corrupted
the water-conservation and campus-metering dashboards.

Per project constraint (2026-04-21): the remediation lives ENTIRELY in the
post-SkySpark processing pipeline. SkySpark itself is treated as an opaque
source; its exported JSON is cleaned downstream before being served to
dashboards.


HOW THIS WAS PROVEN (2026-04-21)
--------------------------------

Dashboard value for bldg 0109E, Feb 2026 = 11,891,829,000 gal.

Arithmetic check:
  0111's raw read on 04/03 (11,891,937)
  minus 0109E's raw read on 02/09 (108)
  times 0109E's multiplier (1000)
  = 11,891,829,000

Line positions in AMRoutputYYYYMMDD.csv:
  02/09 export: 0109E at line 43, 0111 at line 46
  04/03 export: 0109E at line 40, 0111 at line 43    <-- 0111 now sits where 0109E was
  04/21 export: 0109E at line 43, 0111 at line 46

Missing accounts in 04/03 (200015, 200022, 200027) shifted every downstream row
up by three. The importer read row-40's current value and subtracted row-43's
previous value, crossing building identities.


FIX LOCATION (POST-SKYSPARK ONLY)
---------------------------------

The fix is implemented in four layers AFTER data leaves SkySpark. No SkySpark
Folio, Axon, or CSV Importer changes are in scope.

  Layer 1  merge-data.py          validates + merges new SkySpark exports
  Layer 2  _null_outliers.py      20x baseline median guard (Jan-Oct 2025)
  Layer 3  _null_outliers_v2.py   5x pre-gap max guard (Jan-Jul 2025)
  Layer 4  sanitizeOutliers()     in-browser final guard at render time

Layers 2 and 3 run automatically inside update-data.sh after merge, before
push. Layer 4 runs inside the dashboard HTML itself on every page load and
will null any row that slipped past 1-3 -- the belt-and-suspenders line.


THE PIPELINE
------------

update-data.sh runs the following in order:

  1. py merge-data.py data/new/              (existing)
  2. py scripts/sensus-pipeline/_null_outliers.py     (pass 1 sweep)
  3. py scripts/sensus-pipeline/_null_outliers_v2.py  (pass 2, data-gap bumps)
  4. bash push-data.sh                       (existing)
  5. rm data/new/*.json                      (existing)

Each null script writes timestamped .bak files next to the JSON it modifies
and only keeps the backup if rows were actually changed.


WHAT EACH GUARD CATCHES
-----------------------

_null_outliers.py
  Groups rows by (bldgNo, service, meters). Computes the median of positive
  Jan-Oct 2025 values. Nulls any Nov 2025 - Mar 2026 row > 20x that median
  AND > 50,000. Skips groups with fewer than 4 baseline months or median < 100.

_null_outliers_v2.py
  Catches the post-Aug-2025 data-gap bump-ups that the 20x median missed:
  many buildings had Aug-Oct 2025 nulls followed by a giant Nov 2025 catch-up
  value. Nulls any Nov 2025 - Mar 2026 value > 5x the Jan-Jul 2025 max for
  that meter, excluding CHW plants (seasonal by design).

sanitizeOutliers() (in metering.html + water-conservation.html)
  Runtime sweep in the browser. Groups rows by (bldgNo + service + meters),
  takes the Jan-Jul 2025 max, nulls any 2025/2026 value > 20x that max AND
  > 100,000. Console-logs every nulled row under the tag [sanitizeOutliers]
  so the dashboard still shows a gap, not a lie.


VERIFICATION AFTER MONTHLY UPDATE
---------------------------------

1. update-data.sh prints a summary from each null pass. Expect lines like:
     fname.json: nulled N rows; backup -> fname.json.nullscan.YYYYMMDDT...bak

2. Open the dashboards and check the browser console for [sanitizeOutliers]
   warnings. If any appear, layers 2-3 missed an outlier -- investigate the
   source rows in the Sensus AMR export.

3. If new spike shapes appear (e.g., a row-shift pattern that the 20x/5x
   thresholds don't catch), tune the constants in _null_outliers.py and
   _null_outliers_v2.py. The thresholds are intentionally loose enough that
   real peak months survive.
